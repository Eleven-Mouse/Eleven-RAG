# Eleven-RAG 文件解析与检索升级实施落地文档（分阶段）

## 0. 文档说明

- 文档目的：给出可执行的阶段化落地路径，并记录每个阶段的成果与依赖变化。
- 范围：文件解析、切片、持久化、检索链路、入口能力。
- 状态标记：
  - `已完成`：代码已落地并可运行。
  - `待实施`：尚未落地，需要下一阶段执行。

---

## 阶段 0（基线版）`已完成（历史）`

### 完成后实现了什么功能

- 提供最小 RAG API：`/v1/ingest`、`/v1/retrieve`、`/v1/chat`。
- 提供奶龙 CLI 调用入口：`nailong_cli.py`。
- 提供基础导入工具：`nailong_ingest.py`（早期仅 `.md/.txt`）。

### 解决了什么问题

- 从 0 到 1 跑通“导入 -> 检索 -> 问答”主链路。
- 无需前端，也可以在本地命令行完成问答联调。

### 加入了哪些依赖

- `fastapi==0.116.1`
- `uvicorn==0.35.0`
- `pydantic==2.11.7`
- `langchain==0.3.27`

---

## 阶段 1（本地持久化与多格式解析）`已完成`

### 完成后实现了什么功能

1. 文件解析升级：
- 支持 `.md/.txt/.pdf` 导入。
- `.pdf` 使用 `pypdf` 提取可解析文本。
- 相关文件：
  - `eleven-rag/services/document_parser_service.py`
  - `eleven-rag/nailong_ingest.py`

2. 持久化能力落地：
- 元数据持久化到 SQLite：`.rag_store/metadata.db`
- 向量索引持久化到本地 JSON：`.rag_store/vector_index.json`
- 相关文件：
  - `eleven-rag/repositories/metadata_repository.py`
  - `eleven-rag/repositories/vector_repository.py`
  - `eleven-rag/services/container.py`

3. 导入链路改造：
- 文档导入后自动执行“切片 -> 元数据写入 -> 向量索引写入”。
- 同 `document_id` 重新导入会替换旧 chunk，并清理旧索引。
- 相关文件：
  - `eleven-rag/services/ingestion_service.py`

4. 检索链路改造（中文友好）：
- 混合评分：关键词分 + 持久化向量分 + token 向量 fallback。
- 增加宽泛问句兜底（无命中时回退向量结果，最终再回退到最早 chunk）。
- 相关文件：
  - `eleven-rag/services/retrieval_service.py`
  - `eleven-rag/services/text_utils.py`

### 解决了什么问题

- 解决“服务重启后数据丢失”的问题（从内存态变为本地持久化）。
- 解决“宽泛中文问句容易 0 命中”的问题（加入 fallback 策略）。
- 解决“只能导入纯文本文件”的问题（新增 PDF 文本提取能力）。

### 加入了哪些依赖

- 新增：`pypdf==5.9.0`
- 其余依赖保持不变。

## 阶段 2（结构化解析与 API 文件上传）`待实施`

### 阶段完成后将实现什么功能

1. 新增文件上传入口：
- 新增 `POST /v1/ingest/file`（multipart）；
- 支持不经 CLI，直接通过 API 上传文件导入。

2. 结构优先切片：
- 基于标题/段落优先切片，再做长度约束切分；
- 降低“语义被硬截断”问题。

3. 元数据增强：
- 每个 chunk 补充 `page_no/section/source_path/chunk_order/char_count` 等字段；
- 提升引用可追溯性与排错效率。

### 解决什么问题

- 解决“导入入口只在 CLI，不利于系统集成”的问题。
- 解决“固定字符切片导致跨段拼接、语义不完整”的问题。
- 提升引用的可解释性和可审计能力。

### 预计新增依赖

- 可选新增（若采用成熟分块器）：`langchain-text-splitters`
- 若仅自研轻量结构分块：可不新增第三方依赖。

---

## 阶段 3（扫描 PDF OCR 与导入质量门禁）`待实施`

### 阶段完成后将实现什么功能

1. OCR 分支：
- 对扫描型 PDF 在文本提取率低时自动进入 OCR 流程。

2. 导入质量门禁：
- 对每次导入输出质量指标（提取字符数、空 chunk 比例、疑似乱码比例）；
- 低质量导入可告警或拒绝入库。

3. 质量可观测：
- 增加导入日志与失败原因分类（解析失败、OCR失败、空文档等）。

### 解决什么问题

- 解决“扫描 PDF 导入成功但无有效内容”的问题。
- 降低“脏数据入库影响检索与回答质量”的风险。

### 预计新增依赖

- OCR 路线二选一：
  - `rapidocr-onnxruntime`（本地模型）
  - 或 `paddleocr`（能力更全，依赖更重）

---

## 阶段 4（生产化迁移）`待实施`

### 阶段完成后将实现什么功能

1. 存储生产化：
- SQLite -> MySQL（文档/chunk 元数据）
- 本地 JSON 向量索引 -> 独立向量库（如 Qdrant/Milvus）

2. 状态管理生产化：
- 会话与短时状态接入 Redis。

3. 检索增强：
- 接入真实 embedding 模型与独立 reranker。

### 解决什么问题

- 解决“本地持久化不适合多实例/高并发”的问题。
- 提升召回稳定性、可扩展性和线上可运维能力。

### 预计新增依赖

- MySQL 驱动（如 `pymysql` 或 `mysqlclient`）
- 向量库客户端（如 `qdrant-client` / `pymilvus`）
- Redis 客户端（`redis`）
- 可选 rerank/embedding SDK（按选型确定）

## 验收标准（按阶段）

### 阶段 1 验收（当前应已满足）

- `.md/.txt/.pdf` 导入成功；
- 重启服务后检索仍可命中；
- 问句“这份文档讲了什么”可返回引用。

### 阶段 2 验收

- API 文件上传导入成功；
- 结构化切片后，引用片段语义完整性明显提升（人工抽检）。

### 阶段 3 验收

- 扫描 PDF 至少一份可 OCR 成功并可检索；
- 低质量导入能触发告警/拒绝策略。

### 阶段 4 验收

- MySQL/向量库/Redis 全链路联通；
- 同等数据集下召回稳定性与延迟达到目标阈值。
