# AGENTS.md

本文件作用于仓库根目录及全部子目录，作为 Eleven-RAG 的通用执行规则；若子目录存在更细粒度 `AGENTS.md`，则在不冲突前提下共同生效。

## 通用技能入口（using-superpowers）

- 处理任何任务前，先判断当前请求是否适合启用 skill、模板化工作流或专门子 agent。
- 若存在明显匹配的 skill/工作流，优先启用后再继续执行；若当前环境没有可用 skill，则按本文件与子目录 `AGENTS.md` 规则手动执行等价流程。
- 开始动手前，先说明本次采用的执行方式：直接处理、启用 skill，或委派子 agent。
- 若任务较小且没有合适 skill，可直接执行，但仍需先完成目标与约束判断。

## 方案澄清规则（brainstorming）

- 凡是涉及功能新增、行为修改、方案设计、架构调整、规则变更、流程重构，默认先做一次简短澄清。
- 澄清至少覆盖三项：
  1. 目标：要解决什么问题、期望产出是什么；
  2. 约束：时间、范围、兼容性、风险、技术边界；
  3. 备选方案：至少给出一个默认方案；如有明显权衡，补充备选方案与取舍理由。
- 若用户需求已经非常明确且约束充分，可用简短复述代替追问，然后再执行。
- 若关键信息缺失且会影响实现方向，先提问确认，不要直接进入实现。

## RAG 后端专属规则

- 记忆分层固定为三类：偏好记忆、知识记忆、会话记忆；新增能力不得混淆三类职责。
- 新知识处理默认走“对照理解”链路：已知内容、新增信息、潜在混淆点、可类比知识。
- 检索必须采用混合策略，不允许仅依赖单一向量召回；默认为关键词 + 向量 + 重排。
- 回答必须保留来源引用，确保可追溯，禁止无依据补全。
- 记忆数据必须支持可编辑、可删除、可追溯；涉及删除时需考虑索引与缓存一致性。
- 禁止将全部历史会话直接塞入上下文；优先使用检索、摘要和短时状态管理。

## 技术栈约束（MVP）

- 后端框架：`Python + FastAPI + LangChain`
- 结构化数据：`MySQL`
- 向量检索：`独立向量库`（与 MySQL 分离）
- 短时状态与缓存：`Redis`
- 异步链路：`MQ` 为可选后续能力，用于解析、OCR、批处理
- 引入编排框架（如 LangChain）时，仅用于工作流编排和组件集成，不替代核心业务与数据模型。

## 执行要求

- 先思考入口规则，再执行具体操作。
- 优先最小改动，避免未经确认的扩范围设计。
- 对外输出保持简洁，明确说明下一步、风险与结论。
- 涉及检索质量或事实性变更时，优先补充最小可验证测试（召回、重排、引用完整性）。
- 每次完成功能新增、功能修改或架构调整后，必须同步更新根目录 `进度文档.md`，未更新视为任务未完成。
- 更新 `进度文档.md` 时至少包含：`已完成/部分完成/未完成` 状态变化、实现方式变化、当前限制变化、下一步建议（如有）。
