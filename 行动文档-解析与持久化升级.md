# 行动文档：RAG 文件解析与持久化升级（评审版）

## 1. 目标

- 把当前“`.md/.txt + 固定字符切片 + 内存存储`”升级为可持续使用的 RAG 基础能力。
- 支持多格式文档解析（至少 `.md/.txt/.pdf`）。
- 检索层对中文问句更友好，减少“有数据但检索不到”。
- 数据在服务重启后不丢失（持久化）。

## 2. 约束与边界

- 优先最小可运行升级，避免一次性大改。
- 保持现有 API 不破坏：
  - `POST /v1/ingest`
  - `POST /v1/retrieve`
  - `POST /v1/chat`
- 先不做复杂权限体系与分布式部署。
- 回答继续保留来源引用。

## 3. 默认落地方案（分两阶段）

### 阶段 A：本地可持久版本（快速落地）

- Loader/解析：
  - `.md/.txt`：文本 loader。
  - `.pdf`：PDF loader（可先使用非 OCR 方案）。
- 切片：
  - 用 `RecursiveCharacterTextSplitter`，按段落与标点优先切分。
  - 参数默认：`chunk_size=500`、`chunk_overlap=100`（可配置）。
- 向量库：
  - 先落地本地持久化向量库（如 Chroma 持久目录）。
- 元数据：
  - 先用 SQLite（或保留轻量文件索引）记录文档元信息与 chunk 映射。
- 检索：
  - 保留混合检索思路：关键词 + 向量 + 简单重排。
  - 关键词侧补中文友好的分词/字粒度 fallback。

### 阶段 B：生产化版本（你确认后再做）

- 元数据迁移至 MySQL。
- 向量库迁移到独立服务型向量库（如 Milvus/Qdrant）。
- 会话与短时状态迁移到 Redis。
- 重排器替换为可配置 reranker。

## 4. 具体改动清单（文件级）

1. 新增解析与切片模块  
- `eleven-rag/services/document_parser_service.py`  
职责：按文件类型选择 loader，并输出标准 `Document` 列表。

2. 新增向量存储仓储  
- `eleven-rag/repositories/vector_repository.py`  
职责：向量写入、相似检索、按 `document_id` 删除。

3. 新增元数据仓储（阶段 A 先 SQLite）  
- `eleven-rag/repositories/metadata_repository.py`  
职责：文档信息、chunk 索引、来源追溯。

4. 改造导入服务  
- `eleven-rag/services/ingestion_service.py`  
从“纯字符串切片”改为“解析 -> 切片 -> 向量化 -> 元数据写入”。

5. 改造检索服务  
- `eleven-rag/services/retrieval_service.py`  
从内存遍历改为“关键词召回 + 向量召回 + 融合排序”。

6. 保持控制器契约  
- `eleven-rag/controllers/ingestion_controller.py`  
- `eleven-rag/controllers/retrieval_controller.py`  
- `eleven-rag/controllers/chat_controller.py`  
仅调整内部调用，不改接口字段。

7. CLI 工具增强  
- `eleven-rag/nailong_ingest.py` 支持 `.pdf` 并给出清晰失败原因。

8. 配置扩展  
- `eleven-rag/core/config.py` 增加向量库路径、SQLite 路径、切片参数配置。

## 5. 数据流（升级后）

1. 导入：`file_path/content -> parser -> chunks -> embedding -> vector store + metadata store`
2. 检索：`query -> keyword recall + vector recall -> rerank -> topK sources`
3. 问答：`query + sources -> answer(with citations) -> session memory`
4. 删除（预留接口）：`document_id -> metadata delete + vector delete + cache invalidate`

## 6. 兼容与迁移策略

- 保持现有 API 入参不变，避免调用方改造。
- 兼容旧内存流程作为 fallback（可通过配置开关控制）。
- 新增持久目录（如 `.rag_store/`）存储本地向量与 SQLite 文件。

## 7. 风险与应对

1. PDF 质量参差（扫描件）  
应对：先支持可提取文本 PDF；扫描件后续接 OCR 异步任务。

2. 中文检索召回不稳定  
应对：关键词侧增加中文分词/字粒度 fallback；保留向量检索兜底。

3. 本地索引损坏或版本不兼容  
应对：提供重建索引命令（按文档重跑 ingest）。

## 8. 最小验收标准

- 导入 `.md/.txt/.pdf` 至少各 1 份，接口成功并产生 chunk。
- 重启服务后，已导入文档仍可被检索命中。
- 问句“这份文档讲了什么”在有文档情况下能返回至少 1 条引用。
- `chat` 返回包含来源引用（`chunk_id/document_id/score`）。

## 9. 实施顺序与工时预估

1. 阶段 A 基础能力（解析+持久+检索改造）：0.5~1 天
2. 联调与最小测试：0.5 天
3. 阶段 B 生产迁移（MySQL/独立向量库/Redis）：1~2 天（视环境而定）

## 10. 需要你确认的决策

1. 阶段 A 是否接受先用本地持久化（Chroma/SQLite）快速落地？
2. 向量模型先用本地轻量模型还是直接接在线 embedding API？
3. PDF 扫描件是否本轮必须支持 OCR？
